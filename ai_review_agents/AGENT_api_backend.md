### B. 時刻・タイムゾーンの統一（JST厳守）

- [ ] すべてのタイムスタンプがJST（UTC+9）で返されているか
- [ ] `datetime.now()` が `datetime.now(JST)` 等に置き換えられているか（UTC誤差なし）
- [ ] 予報データの「today」判定がJST基準か（UTCで判定すると9時間ずれる）
- [ ] ログ出力の時刻もJSTか

### C. エラーハンドリングの品質

- [ ] Open-Meteo API タイムアウト（例: 10秒）が設定されているか
- [ ] Open-Meteo が落ちている場合に500ではなく503を返し、フロントに伝わるか
- [ ] 不正なJSONリクエストに400を返すか（500にならないか）
- [ ] `spot_name` に特殊文字（`../`、`<script>` 等）を渡したときにSQLインジェクション・パストラバーサルが起きないか

### D. Renderデプロイ固有の確認

- [ ] `Procfile` の起動コマンドが正しいか（`gunicorn start:app` 等）
- [ ] 環境変数（APIキー等）がRenderのEnvironment Variableで設定されているか（コードにハードコードされていないか）
- [ ] Renderの無料プランのスリープ（15分無操作でスリープ）に対する対策があるか（ヘルスチェック等）
- [ ] デプロイ後に旧サービス（`rishiri-kelp-forecast`、v2.0.0）が完全に削除されているか

### E. レスポンス品質

- [ ] `GET /api/forecast` のレスポンスに `shortwave_radiation` が含まれているか（CLAUDE.mdの要件）
- [ ] `precipitation_probability`（降水確率）が含まれているか（CLAUDE.mdのロードマップ）
- [ ] `dewpoint_2m`（露点）が含まれているか（霧リスク計算の前提）
- [ ] レスポンスのContent-Typeが `application/json; charset=utf-8` になっているか
