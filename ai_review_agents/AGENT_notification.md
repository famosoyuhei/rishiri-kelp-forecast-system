## チェックリスト

### A. 3種類の通知の発火条件

**夕方通知（毎日 16:00 JST）:**
- [ ] 16:00 JSTに正確に発火するか（UTCとの混同がないか）
- [ ] 翌日の乾燥予報を `tomorrowUTC` で正しく参照しているか
- [ ] 翌日が沖止め日に設定されている場合、通知がスキップされるか

**早朝通知（毎日 01:30 JST）:**
- [ ] 01:30 JSTに正確に発火するか
- [ ] `scheduleEarlyMorningNotification` 関数が正しくスケジュールされているか
- [ ] 当日の乾燥予報を正しく参照しているか

**緊急アラート（随時・1時間ポーリング）:**
- [ ] 1時間ごとにAPIを呼び出しているか（ポーリング間隔の確認）
- [ ] スコアが30点以上変化したときのみ通知が発火するか（閾値の確認）
- [ ] 30点未満の変化では通知が出ないか（誤報防止）

### B. 統一ゲーティング関数 `shouldSendNotification(targetDate)`

- [ ] この関数が **すべての通知発火前** に必ず呼ばれているか（バイパスがないか）
- [ ] 沖止め日設定を正しくチェックしているか
- [ ] シーズン範囲（6〜9月デフォルト）外の日付に通知しないか
- [ ] ON/OFFが「オフ」の干場に通知が送られていないか

### C. 通知設定UI

- [ ] 免責事項同意フロー（初回のみ）が表示されるか
- [ ] 干場ごとの通知ON/OFF切替が機能するか
- [ ] 通知音（Web Audio API）のON/OFFが設定できるか
- [ ] バイブレーション設定のON/OFFが設定できるか
- [ ] テスト送信ボタンで即時通知が届くか
- [ ] 沖止め日の設定（カレンダーUIまたは日付入力）が正常に動作するか

### D. 通知履歴

- [ ] 通知履歴が最大50件保持されるか（51件目で古い方が消えるか）
- [ ] `notificationData` が `hoshibaNotificationData` キーでlocalStorageに保存されているか
- [ ] 通知履歴画面で過去の通知内容・時刻が確認できるか
- [ ] 履歴の通知をタップ/クリックで対象干場の詳細に遷移できるか

### E. エラー・エッジケース

- [ ] ブラウザの通知権限が「拒否」の場合に、設定変更を促すメッセージが表示されるか
- [ ] 通知権限が「未設定」の場合に許可リクエストが正しく表示されるか
- [ ] APIが取得できない（オフライン）場合に、通知スケジューリングが壊れないか
- [ ] 日付をまたぐタイミング（23:59 → 0:00 JST）でスケジューラが正しくリセットされるか

## 報告形式

```
【重大度】🔴高 / 🟡中 / 🟢低
【通知種別】夕方 / 早朝 / 緊急アラート / 設定UI / 履歴
【該当箇所】ファイル名:行番号（例: kelp_drying_map.html:3456）
【問題内容】何が起きているか
【再現手順】どうすれば確認できるか
【修正提案】どう直せばよいか
```

## 精査の観点

> 「通知が来なかった」は漁師さんにとって機会損失です。  
> 逆に「通知が来すぎる」と迷惑です。  
> 「必要なときだけ・確実に」届くことが最重要です。
